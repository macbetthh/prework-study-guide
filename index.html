Prework Study Guide

HTML

  • HTML = Hypertext Markup Language
  • Made up of elements like text, images, headers, and lists.
  • Meta-related, content divisions, and semantic elements divide and describe pages.
  • Always remember to create a sturdy "frame" using HTML.
  • The head element contains information about the webpage. This includes the title tag which names the page.
  • The body element represents the visible content shown to the user.
  • DOCTYPE isn't case senstive -- it's a declaration. HTML elements ARE case-sensitive.

CSS

  • CSS = Cascading Style Sheets
  • Programming language used by developers to define how webpage content is presented to users
  • CSS defines the colors, aesthetics, visual layout, and organization

CSS Styling


  1. inline CSS -- used to style an element directly in the HTML file
    • pro: easy and quick styling
    • con: must be edited individually whenever designs need change
    • example:
      <h1 style="color:red;">Prework Study Guide</h1>

  2. internal CSS style sheet -- can be embedded directly in the HTML file using a <style> element in the <head> element
    • CSS rules only apply to the page it's tied to -- a signle change to a rule will apply to all elements tagged with that rule
    • CSS classes and IDs can style multiple elements in the page code
    • example -- this code states that all h1 elements on its webpage will be red:
      <head>
          <style>
            h1 {
              color: red;
            }
          </style>
      </head>

  3. external CSS style sheet -- most common way to style a webpage
    • preferred in most cases because it allows developers to keep all their CSS rules in a separate file making design changes easier
    • Separation of concerns is a widely used term in web dev and refers to separating code into diff files based on use
    • example of stylesheet linking:
      <head>
         <link rel="stylesheet" href="./assets/style.css" />
      </head>
  • margin indicates how much space we want around the outside of an element.
  • padding indicates how much space we want around the content inside an element.
  • The box-shadow CSS property adds shadow effects around an element's frame
  • The border property assigns a border to the specified class/id (example: the 1px solid gray borders in each section of this page)
  • The box-shadow property adds shadow effects (see the bottom of each box on this page) -- we haven't "learned" this yet, but I've used these before and found a style I liked to spruce up the basic styling we've started so far.
  • Assigning colors in CSS can be done in many ways, including by semantic references such as "gray" or "blue". We can also refer to a color by its hexadecimal code, or hex #.

Git

  • Some Bash Commands:
    • ls -- shows list of files & directories on comp
    • cd-- change directories
    • pwd -- print working directory
    • mkdir -- creates new directories
    • touch -- creates new file inside directory
    • start -- opens file

  • Repositories -- folders to store files that contain code, to store files that contain other info or README, and to store version history for all files
    • Naming conventions:
      • Be unique
      • All lowercase
      • Use dashes not spaces
      • Be consistent

  • Branches
    • main -- default branch
    • feature branch -- copy of main - push up changes by opening a pull request
    • pull request -- used to merge a comparison branch into the base branch
      • Git PR command pieces:
        • git -- precedes every Git Statement
        • checkout -- move working branch to new branch
        • -b -- flag creates new branch
        • feature/starter-code -- new branch name for project example - also describes type of branch (feature)
        • git status -- branch you're currently in
        • -A -- add all changes
        • add -- adds modifications to current branch
        • git commit -m "add starter code" -- git command we used to create a snapshot of currently staged changes to our feature branch -- (the -m flag associates a message with the commit)
        • pull -- receive a branche's modifications into the local envrionment
        • origin -- the source the pull will be in Github repo
        • push -- push changes we have locally to our remote Github branch
